4 layouts para a documentação
do Arcana DataTable

A docs atual é uma landing de scroll único — boa vitrine, fraca como documentação de trabalho. Cada proposta abaixo ataca um jeito diferente de usar a doc. Elas não são excludentes: dá para combinar (ex.: Trilha no "Começando" + Manual como referência).

hoje: single-page editorial conteúdo real da API navegue com a barra 01–04
PROPOSTA 01

Manual — referência três painéis

Estilo Stripe: nav com scroll-spy, prosa no centro, código sticky à direita sempre sincronizado com a seção lida. Toggle global React/Vue.

Melhor para: consulta diária de quem já integrou a lib. Atenção: menos "wow"; exige manter prosa e código pareados.
Abrir protótipo →
PROPOSTA 02

Bancada — playground de props

Estilo Storybook: knobs ligam/desligam props, o preview e o código gerado reagem na hora. A config é a própria documentação.

Melhor para: descobrir recursos experimentando, sem ler. Atenção: mais custo de manutenção; prosa conceitual fica curta.
Abrir protótipo →
PROPOSTA 03

Trilha — guia em capítulos

Curso curto e linear: 8 capítulos da instalação à tabela completa, com rail de progresso que preenche no scroll e objetivo declarado por capítulo.

Melhor para: onboarding de quem nunca viu a lib. Atenção: ruim para consulta pontual; precisa de uma referência ao lado.
Abrir protótipo →
PROPOSTA 04

Cookbook — receitas buscáveis

"Quero fazer X": busca + facetas sobre 14 cards-receita em vernáculo de planilha (coordenadas, seleção estilo célula), com drawer de preview + código.

Melhor para: quem já usa e quer a receita pronta em segundos. Atenção: não ensina conceitos (remote vs dataset) — pressupõe contexto.
Abrir protótipo →
Como avaliar: abra cada proposta e tente responder três perguntas — "como instalo?", "como faço filtro por lista?" e "que métodos o controller tem?". O layout vencedor é o que responde mais rápido. Todo o conteúdo vem da API real (docs/src/App.tsx); as tabelas das demos são estáticas.
